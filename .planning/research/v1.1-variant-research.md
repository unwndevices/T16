# v1.1 Variant Support — Research Digest

**Source:** Curated from `feature/hw-abstraction-layer` design docs (Aug 2025, ~4000 lines across `claude/hardware-abstraction-variants.md`, `forward-compatible-configuration.md`, `configuration-migration-system.md`, `project-outline.md`, `implementation-phases-tracker.md`).

**Status:** Pre-discuss research. The original branch is based on the pre-v1.0 codebase and cannot be merged. This digest extracts the design ideas worth carrying forward into v1.1; everything else from those docs is intentionally discarded.

**Validated reference:** Branch `origin/3dot0` (commit `f0a8eba`, Nov 2024) booted T32 hardware on the same physical unit currently on hand. Its hardware facts are empirically verified.

---

## Scope

v1.1 delivers compile-time variant support (T16 + T32) on top of the v1.0 firmware and editor-tx architecture. T64 is explicitly out of scope — no hardware exists.

The web editor (`editor-tx/`) is already at quality bar: TypeScript, React 19, Radix design system, transport-agnostic `protocol/sysex.ts`, ajv-based `services/configValidator.ts`, BLE transport. Variant support extends these existing seams; it does not require a rewrite.

---

## Hardware abstraction — design to adopt

### Compile-time variant selection
- PlatformIO build flag (`-DT16` or `-DT32`) selects variant at compile time.
- Per-variant pinout header: `pinout_t16.h`, `pinout_t32.h`.
- Single `pinout.h` aliases `namespace CurrentPinout = T16Pinout` (etc.) based on the build flag.
- Three build envs in `platformio.ini`: `t16_debug`, `t16_release`, `t32_debug`, `t32_release`.

### `HardwareVariantConfig` constexpr carrier
Compile-time struct exposing:
- `TOTAL_KEYS` (16 / 32)
- `MUX_COUNT` (1 / 2)
- `LED_COUNT` = `TOTAL_KEYS` (T16: 16, T32: 32 — one LED per key, confirmed against `3dot0`)
- Capability flags: `HAS_TOUCH_SLIDER` (true for both), `SUPPORTS_KOALA_MODE` (likely T16-only — confirm in discuss)

v1.0's `Adc`, `Keyboard`, `LedManager` classes consume these constants directly, replacing fixed macros (`BANK_AMT`, `NUM_LEDS`).

### `MultiplexerConfig` struct
Per-mux entry, repeated `MUX_COUNT` times:
```cpp
struct MultiplexerConfig {
    uint8_t commonPin;        // unique per mux
    uint8_t enablePin;        // 255 = always on
    uint8_t selectPins[4];    // shared S0..S3 across all muxes
    uint8_t keyMapping[16];   // physical channel -> logical key index
    bool useSharedSelect;
};
```

Matches `3dot0`'s validated `adc_config.InitMux(PIN_COM, PIN_S0, PIN_S1, PIN_S2, PIN_S3, 2)` — shared select pins, separate common pins per mux.

### Validated T32 key permutation (from `3dot0`)
```cpp
Key keys[] = {14, 15, 13, 12, 30, 31, 29, 28,
              10, 11,  8,  9, 26, 27, 24, 25,
               1,  0,  3,  2, 17, 16, 19, 18,
               5,  4,  7,  6, 21, 20, 23, 22};
```
Decomposes naturally into two 16-entry `keyMapping[]` arrays — the first 16 indices for mux 0 (mirrors T16 layout), the second 16 for mux 1.

### Scan optimization
Batch all muxes per channel: set `S0..S3`, then read every `commonPin` before incrementing the channel. With shared select pins this is one settle per channel × `MUX_COUNT` reads, instead of one settle per (mux × channel).

## Config schema — design to adopt

### Add `variant` discriminator
JSON config gains a top-level `"variant": "T16" | "T32"` field. Required from vNext onwards.

### Bump version
v103 → v104 (or v200 if we want a major break — decide in discuss). Single migration rule, default-injection for missing fields.

### Per-key array sizing
Arrays sized by `TOTAL_KEYS` are variant-bound. The handling of cross-variant loads (T16 config on T32 device, or vice-versa) is an open question; the conservative default is "extend with defaults / truncate" with a user-visible warning, not silent reject.

### Optional on-disk header
Cheap robustness: prefix the LittleFS JSON with `magic + version + size + checksum` bytes. ArduinoJson reads/writes the body untouched. Skip if it complicates the existing `DataManager` template path.

### Migration as small versioned rules
Concept only — a single rule for v103 → vNext is sufficient. Reject the doc's `UniversalConfiguration` + Dijkstra path-finding machinery; ArduinoJson + a hand-written transform function is enough.

### Calibration file portability
`/calibration_data.json` is sized by `TOTAL_KEYS`. Tag with variant; force recalibration on mismatch.

## Editor-tx (web) — design to adopt

The web editor's existing seams accommodate variants without rework:
- `protocol/sysex.ts` is transport-agnostic — adds variant-aware payload sizing.
- `services/configValidator.ts` (ajv) — schema gains `variant` enum + per-variant array sizes.
- `contexts/ConfigContext.tsx` — variant lifted to context state, derived from device handshake or loaded config.
- Components branch on `variant` for: 4×4 vs 4×8 keyboard layout, calibration view, per-key arrays.
- `design-system/` is untouched.

Runtime variant detection comes from a SysEx handshake response (the firmware reports its compiled variant). The flasher (`pages/Upload.tsx` + esptool-js) needs to choose the right `.bin` — open question whether by user selection or auto-detect via existing connection.

## Explicitly NOT adopted

The following from the source docs are scope creep, premature abstraction, or duplicative of v1.0's architecture. Listed so they don't sneak back in during discuss:

- **T64 variant** (no hardware, speculative pinouts)
- **`ExtensibleConfiguration` TLV/extension blocks** (JSON is already extensible)
- **`ParameterRegistry` + `ParameterDescriptor`** (duplicates editor-tx's ajv on-device; burns flash)
- **`FeatureFlagManager` 64-bit bitmask** with future-feature enums like `MIDI_2_0_SUPPORT`, `CLOUD_PRESETS`, `AUDIO_EFFECTS`
- **`UniversalConfiguration` heap-allocated key-value bag** (ArduinoJson covers it)
- **Reversible migrations / downgrade path** (no user need)
- **New binary SysEx protocol** (v1.0 standardized on JSON-over-SysEx; don't entangle)
- **Triple-redundancy + CRC32 critical-data protection** (solving a non-observed problem)
- **The doc's web-rewrite plan** (editor-tx is already at quality bar)
- **Doc's `static constexpr auto MUX_CONFIGS = CurrentPinout::MUX_CONFIGS` aliasing** in headers (ODR risk, already a known issue per CLAUDE.md)

## Open questions (resolve in `/gsd-discuss-phase`)

1. **Variant field semantics.** Required vs optional? Loading T16 config on T32 device — extend with defaults, reject, or prompt?
2. **Firmware distribution.** Three `.bin` files — does the web flasher auto-detect target variant, or does the user pick? `3dot0` proves T32 boots; what's the upgrade path for users currently on `main` T16?
3. **v103 → vNext field diff.** Need to enumerate the actual delta between current `Configuration.hpp` structs and any new variant-aware fields.
4. **Per-key array extension semantics.** When the schema grows from 16 to 32 entries, do we pad with defaults, mirror the first 16, or tile? Affects editor UX (does the user see "this T16 preset extended to T32"?).
5. **Calibration file format.** Variant tag + force-recalibrate on mismatch, or migrate?
6. **Koala mode (and any other T16-shaped features) on T32** — disabled, redesigned, or not surfaced in the editor UI?
7. **Namespace style.** Doc uses `T16Pinout::`; v1.0 has no namespaces. Accept scoped exception for pinouts, or stick to `T16_` prefix consistency?
8. **T64 placeholder.** Reserve enum value to avoid renumbering later, or keep variant set strictly `{T16, T32}`?

## Suggested phase shape (rough — refine in plan-phase)

1. Per-variant `pinout_t16.h` / `pinout_t32.h` + `pinout.h` aliasing + `platformio.ini` envs. Builds for both variants pass; T16 still works.
2. `HardwareVariantConfig` adoption — `Adc`, `Keyboard`, `LedManager`, `DataManager` consume `TOTAL_KEYS` / `MUX_COUNT` / `LED_COUNT` from the variant config instead of macros. T16 behavior unchanged.
3. T32 hardware bring-up — multi-mux ADC scan, dual-mux init, key permutation, calibration on physical T32. Validate against `3dot0` reference behavior.
4. Config schema v104 — `variant` field + migration from v103 + calibration file variant tagging. T16 configs migrate cleanly.
5. Editor-tx variant awareness — `ConfigContext` variant state, ajv schema update, conditional UI for keyboard/calibration views, flasher variant selection.

Each phase ends with both T16 and T32 functional; nothing branches off into long-lived parallel state.

# Plan 12.02 — T32 key permutation in variant_t32.hpp keyMapping[]

## Objective

Replace the placeholder identity-permutation `keyMapping` arrays in `src/variant_t32.hpp` with the validated 32-key permutation derived from `origin/3dot0:src/main.cpp:10`. Implements **D12.3** and addresses requirement **T32-03**.

## Background

Ground truth (origin/3dot0:src/main.cpp lines 9–13):

```c++
#ifdef T32
Key keys[] = {14, 15, 13, 12, 30, 31, 29, 28, 10, 11, 8, 9, 26, 27, 24, 25, 1, 0, 3, 2, 17, 16, 19, 18, 5, 4, 7, 6, 21, 20, 23, 22};
#endif
```

In 3dot0, `Key(uint8_t index)` stores `mux_idx = index`, and `_channels[mux_idx + 16*muxNumber]` is the ADC slot read. The new architecture (Phase 11) replaces the per-key `mux_idx` with a per-channel `keyMapping[ch]` lookup that translates **scan channel → logical key index**. This is the inverse of the 3dot0 form.

### Inversion procedure

The 3dot0 array partitions naturally into 8 groups of 4 (= 8 columns × 4-key chunks). Indices 0–15 belong to mux 0; indices 16–31 belong to mux 1. The mux assignment of each group of 4 in declaration order is:

| 3dot0 group | Indices | Mux | Notes |
|---|---|---|---|
| 0 | 14, 15, 13, 12 | 0 | logical keys at electrical columns 0–3 |
| 1 | 30, 31, 29, 28 | 1 | |
| 2 | 10, 11,  8,  9 | 0 | |
| 3 | 26, 27, 24, 25 | 1 | |
| 4 |  1,  0,  3,  2 | 0 | |
| 5 | 17, 16, 19, 18 | 1 | |
| 6 |  5,  4,  7,  6 | 0 | |
| 7 | 21, 20, 23, 22 | 1 | |

**Mux 0 ordered (read order = scan channel 0..15 = `iterator`):**
The 3dot0 reader writes `_channels[iterator]` for mux 0 and `_channels[iterator + 16]` for mux 1, with `iterator` running 0..15. The mapping `keys[]` then consumes `_channels` *by index* — `keys[0]` reads from `_channels[mux_idx=14]`. Equivalent statement: physical column scanned at iterator `i` is the column whose logical key index is `mux_0_array[i]`.

To recover `mux_0_array[i]` (and `mux_1_array[i]`), the inversion reads:

- The 3dot0 `keys[]` pairs every physical-key-position `pos` (= 0..31, the array index) with the channel it reads from (`mux_idx = keys[pos]`). Inverting: given channel `c` (0..31), what physical position `pos` is wired to it? That gives `keyMapping[ch] = pos` where `pos < 16` for mux 0 and `pos ≥ 16` for mux 1, but `keyMapping` returns a **logical key index**, not a physical position.
- Logical key index = physical position (the new architecture stores logical = physical position; LED mapping and matrix layout consume the same index). So the inversion table directly yields `keyMapping`.

**Computed result:**

For each electrical channel `c ∈ [0, 32)`:
- Mux 0 (`c ∈ [0, 16)`): the physical key position `pos` such that `keys[pos] == c`.
- Mux 1 (`c ∈ [16, 32)`): the physical key position `pos` such that `keys[pos] == c`. Store `pos` in `mux_1.keyMapping[c - 16]`.

Inverting the 3dot0 array yields:

```
mux_0_keyMapping (channel 0..15  -> logical key index):
  17, 16, 19, 18,  25, 24, 27, 26,  10, 11,  8,  9,   2,  3,  0,  1

mux_1_keyMapping (channel 0..15  -> logical key index):
  21, 20, 23, 22,  29, 28, 31, 30,  14, 15, 12, 13,   6,  7,  4,  5
```

(Derivation walkthrough is reproduced in this plan's tests so a reviewer can audit it; see Task 12.02.2.)

<task id="12.02.2" type="execute">
  <action>
    Add a static-assert validation block at the bottom of `src/variant_t32.hpp` (inside `namespace variant::t32`, after `kMuxes` definition) that mechanically checks the inversion against the 3dot0 array. This catches typos and serves as in-source documentation.

    ```cpp
    namespace detail {
      // Verbatim from origin/3dot0:src/main.cpp:10 — DO NOT EDIT.
      inline constexpr uint8_t k3dot0Keys[32] = {
          14, 15, 13, 12, 30, 31, 29, 28,
          10, 11,  8,  9, 26, 27, 24, 25,
           1,  0,  3,  2, 17, 16, 19, 18,
           5,  4,  7,  6, 21, 20, 23, 22,
      };

      // Compile-time inversion: for each channel c, find pos such that k3dot0Keys[pos] == c.
      // Mux 0 owns c ∈ [0,16); Mux 1 owns c ∈ [16,32).
      constexpr uint8_t InvertedKeyMapping(uint8_t channel) {
          for (uint8_t pos = 0; pos < 32; ++pos) {
              if (k3dot0Keys[pos] == channel) return pos;
          }
          return 0xFF;
      }

      // Verify Mux 0 keyMapping matches the inversion.
      static_assert([] {
          for (uint8_t ch = 0; ch < 16; ++ch) {
              if (kMuxes[0].keyMapping[ch] != InvertedKeyMapping(ch)) return false;
          }
          return true;
      }(), "T32 mux 0 keyMapping does not match origin/3dot0 inversion");

      // Verify Mux 1 keyMapping matches the inversion (channels 16..31).
      static_assert([] {
          for (uint8_t ch = 0; ch < 16; ++ch) {
              if (kMuxes[1].keyMapping[ch] != InvertedKeyMapping(ch + 16)) return false;
          }
          return true;
      }(), "T32 mux 1 keyMapping does not match origin/3dot0 inversion");
    } // namespace detail
    ```

    This is the canonical test for T32-03 — if the build succeeds with -DT32, the permutation is correct by construction. No separate `test_native` env is required. (Optional: also write the host unit test in 12.02.3 for non-PlatformIO reviewers.)

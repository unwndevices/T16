<objective>
Wire the firmware's `CMD_CAPABILITIES` handshake (delivered in Phase 11) into editor-tx so the connected device's `variant` and `capabilities` reach `ConfigContext` automatically. This closes EDITOR-01's "derived from device SysEx handshake response" requirement.
</objective>

<must_haves>
  <truths>
    - The protocol command byte is `0x07` (matches `src/SysExProtocol.hpp` line 16: `constexpr uint8_t CMD_CAPABILITIES = 0x07`). Editor-side constant MUST equal firmware constant — there is no negotiation step.
    - The firmware response payload is ASCII JSON starting at byte index 3 of the SysEx data buffer (after `[CMD_CAPABILITIES, SUB_RESPONSE, status]`), and the JSON shape is exactly: `{"variant":"T16"|"T32","capabilities":{"touchSlider":bool,"koalaMode":bool}}` (per `src/SysExHandler.cpp` lines 290+).
    - The capabilities request is sent ONLY after the version-response handler has fired successfully — this proves MIDI plumbing is up and avoids dropping the response (Research §12 race-condition mitigation).
    - On parse failure the connection does NOT die — log a warning, leave `isHandshakeConfirmed = false`, fall back to the existing offline behavior (`FALLBACK_CAPABILITIES[variant]` from Plan 14-01).
    - `setVariant()` is called BEFORE `setCapabilities()` so the variant transition is atomic from a render perspective: the variant changes, then capabilities arrive in the same effect tick.
    - The capabilities call passes `fromHandshake: true` so `isHandshakeConfirmed` flips to true in ConfigContext (per Plan 14-01 Task 3).
    - This plan does NOT modify firmware. The firmware-side implementation landed in Phase 11 (Plan 11-05).
  </truths>
  <nonneg>
    - No new design tokens, no UI changes in this plan.
    - `npm run typecheck` and `npm run test` clean after this plan.
    - The version-handshake code path remains unchanged for any error case — capabilities is purely additive.
  </nonneg>
</must_haves>

    3. Add the predicate near `isConfigResponse`:

    ```ts
      return parsed !== null && parsed.cmd === CMD.CAPABILITIES && parsed.sub === SUB.RESPONSE
    }
    ```
    (If `parseSysExMessage` does not currently return a typed object exposing `cmd`/`sub`, mirror the predicate style of `isConfigResponse` exactly — read `isConfigResponse` first to see the actual signature.)

    4. Add the JSON parser:
    ```ts
    /**
     * Parse the JSON payload from a CMD_CAPABILITIES response.
     * Layout: bytes [0]=CMD, [1]=SUB, [2]=status, [3..]=ASCII JSON.
     * Returns null on any parse failure or unknown variant string —
     * caller treats null as "handshake unavailable, use fallback".
     */
    export function parseCapabilitiesPayload(
      data: Uint8Array,
    ): { variant: Variant; capabilities: Capabilities } | null {
      if (data.length < 4) return null
      try {
        const json = String.fromCharCode(...data.slice(3))
        const parsed: unknown = JSON.parse(json)
        if (
          typeof parsed !== 'object' ||
          parsed === null ||
          !('variant' in parsed) ||
          !('capabilities' in parsed)
        ) return null
        const v = (parsed as { variant: unknown }).variant
        const c = (parsed as { capabilities: unknown }).capabilities
        if (!isVariant(v)) return null
        if (
          typeof c !== 'object' ||
          c === null ||
          typeof (c as { touchSlider?: unknown }).touchSlider !== 'boolean' ||
          typeof (c as { koalaMode?: unknown }).koalaMode !== 'boolean'
        ) return null
        return {
          variant: v,
          capabilities: {
            touchSlider: (c as { touchSlider: boolean }).touchSlider,
            koalaMode: (c as { koalaMode: boolean }).koalaMode,
          },
        }
      } catch {
        return null
      }
    }
    ```

    Use `CMD` and `SUB` imports already present in the file. Do not duplicate imports.

| T-14-02-01 | Tampering / injection | Capabilities JSON payload from device | mitigate | `parseCapabilitiesPayload` strict-validates: variant via `isVariant`, capabilities boolean fields type-checked; unknown shapes return null and are logged. No `eval`, no dynamic property access on untrusted keys. |
| T-14-02-02 | Denial of service | Malformed JSON crashing connection handler | mitigate | Parser wraps `JSON.parse` in try/catch and returns null. Handler logs warning and continues without applying capabilities — connection stays alive. |
| T-14-02-03 | Spoofing | Hostile MIDI device claiming to be T32 to flip UI | accept | The capabilities response only changes editor display state; it cannot trigger destructive actions (flashing requires explicit user click in Plan 14-05). User has physical control of which device is connected. |
| T-14-02-04 | Information disclosure | console.warn output of malformed payload bytes | mitigate | Warning message contains only the literal `'capabilities payload unparseable'` — no payload bytes printed. Diagnostic info stays out of console logs. |

Severity: all LOW. The only HIGH-impact risk in editor-tx (firmware flashing) is gated by Plan 14-05's explicit-confirmation modal, which does not depend on this plan's data.

**1. Replace the existing `firmware-build` job entirely with this matrix-strategy version:**

```yaml
  firmware-build:
    runs-on: ubuntu-latest
    strategy:
      fail-fast: false
      matrix:
        env: [t16_debug, t16_release, t32_debug, t32_release]
    steps:
      - name: Checkout
        uses: actions/checkout@v4

      - name: Cache PlatformIO
        uses: actions/cache@v4
        with:
          path: ~/.platformio
          key: pio-${{ hashFiles('platformio.ini') }}-${{ matrix.env }}
          restore-keys: |
            pio-${{ hashFiles('platformio.ini') }}-
            pio-

      - name: Set up Python
        uses: actions/setup-python@v5
        with:
          python-version: '3.11'

      - name: Install PlatformIO
        run: pip install platformio

      - name: Build firmware (${{ matrix.env }})
        run: pio run -e ${{ matrix.env }}
```

Key changes vs. current job:
- Add `strategy.fail-fast: false` so a T32 regression doesn't cancel T16 jobs (and vice versa).
- Add `strategy.matrix.env` listing all four envs.
- Cache key now includes `${{ matrix.env }}` so each matrix leg has its own .pio/build cache (faster incremental builds).
- The `pio test -e native` step is REMOVED from this job — it moves to a new `firmware-tests` job below.
- Replace `pio run -e esp32s3` with `pio run -e ${{ matrix.env }}`.

**2. Add a NEW `firmware-tests` job immediately after `firmware-build`:**

```yaml
  firmware-tests:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout
        uses: actions/checkout@v4

      - name: Cache PlatformIO
        uses: actions/cache@v4
        with:
          path: ~/.platformio
          key: pio-${{ hashFiles('platformio.ini') }}-native
          restore-keys: pio-${{ hashFiles('platformio.ini') }}-

      - name: Set up Python
        uses: actions/setup-python@v5
        with:
          python-version: '3.11'

      - name: Install PlatformIO
        run: pip install platformio

      - name: Run firmware unit tests
        run: pio test -e native
```

This job is independent of the matrix — it tests the `[env:native]` Unity suite once.

<task type="checkpoint:human-verify" gate="blocking">
  <name>Task 2: Manual T16 hardware smoke test</name>
  <what-built>
Phase 10 has refactored the build system and pinout headers without changing any runtime behavior on T16 hardware:
- Plan 01 split the build envs (T16/T32 × debug/release).
- Plan 02 introduced namespaced `pinout::t16` constexpr constants.
- Plan 03 migrated every call site from `PIN_*` macros to `CurrentPinout::*`.
- Plan 04 (this plan) updated CI to verify all four envs build green.

ROADMAP success criterion #4 demands: "Existing T16 firmware behavior is identical after the rename (smoke test on hardware)." This task verifies that on a physical T16 device.
  </what-built>
  <how-to-verify>
1. Build the T16 release binary locally:
   ```bash
   pio run -e t16_release
   ```
   Expected: exits 0, produces `.pio/build/t16_release/firmware.bin`.

2. Flash the binary to a physical T16 device:
   ```bash
   pio run -e t16_release -t upload
   ```
   (Or use the editor-tx web flasher if you prefer the standard release path.)

3. Power-cycle the device and verify the **startup animation** plays correctly (LED matrix sweep + slider strip animation).

4. Connect a USB-MIDI host (DAW or MIDI monitor app like MIDI Monitor, MIDI-OX, or `aseqdump`) and verify:
   - **Keyboard mode (default):** Pressing each of the 16 keys produces a NoteOn / NoteOff with the expected note number for the active bank's keyboard scale. All 16 keys produce output (no dead keys).
   - **Touch slider:** Sliding a finger up/down the touch strip generates CC messages that smoothly traverse the configured CC range (default Pitch Bend or whatever the active config sets).
   - **Mode button:** Pressing the MODE button (PIN_MODE = 0) cycles through modes (Keyboard → Strum → XY-Pad → ... → Keyboard). LED feedback should match each mode's signature pattern.
   - **Touch button:** Pressing the TOUCH button (PIN_TOUCH = 21 on REV_B) registers a mode-or-bank action depending on current state.

5. Verify there is **no behavioral regression** vs. a known-good v1.0 binary. If you have a reference recording of MIDI output from before this phase, compare. Otherwise rely on subjective "feels identical."

6. If TRS MIDI output is part of your setup, verify TRS MIDI also still produces messages (PIN_TX2 / PIN_TX migration in Plan 03 covered this).

If everything behaves identically to pre-Phase-10 firmware → approve. If anything is wrong (a key is dead, slider direction inverted, LED wrong color, MIDI silent, MODE button not advancing modes, etc.) → describe the symptom; the most likely cause is a swapped pin in the migration (e.g., S0 ↔ S1 typo in `pinout_t16.h`) or a missed call site.
  </how-to-verify>
  <resume-signal>Type "approved" if hardware behavior is identical to pre-Phase-10 firmware. Otherwise describe the symptom and which input is misbehaving so the failure can be traced back to the migrated pin.</resume-signal>

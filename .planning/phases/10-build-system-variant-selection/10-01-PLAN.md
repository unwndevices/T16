<objective>
Replace the existing `[env:esp32s3]` and `[env:release]` with four explicit envs (`t16_debug`, `t16_release`, `t32_debug`, `t32_release`) that select the variant via `-DT16` / `-DT32` build flags. Set `default_envs = t16_release` so bare `pio run` continues to build T16 release. Preserve `[env_common]` (shared `lib_deps`, framework, board settings) and leave `[env:native]` untouched.

Purpose: Make variant selection explicit and matrixable in CI (per D10.1, D10.2, D10.4). This plan does NOT change pinout headers or any source files — those land in Plan 02. Until Plan 02 merges, the new envs will fail to compile (pinout.h still emits unconditional `#define`s and doesn't handle `-DT16`/`-DT32`); this is acceptable because Plan 01 and Plan 02 are in the same wave and committed together before CI runs (see <verification> note).

Output: A modified `platformio.ini` with four firmware build envs, `default_envs` pointing at `t16_release`, and the existing `[env_common]` and `[env:native]` sections preserved.
</objective>

```ini
	${env_common.build_flags}
	-Os
	-DCORE_DEBUG_LEVEL=0
	-DREV_B
	-DT16

[env:t32_debug]
extends = env_common
board = unwn_s3
build_flags =
	${env_common.build_flags}

	-DCORE_DEBUG_LEVEL=5
	-DREV_B
	-DT32

[env:t32_release]
extends = env_common
board = unwn_s3
build_flags =
	${env_common.build_flags}
	-Os
	-DCORE_DEBUG_LEVEL=0
	-DREV_B
	-DT32
```

5. Leave `[env:native]` completely untouched (unit-test environment).

6. Use TAB indentation inside `build_flags` (matches existing file style — verify by reading the current file).

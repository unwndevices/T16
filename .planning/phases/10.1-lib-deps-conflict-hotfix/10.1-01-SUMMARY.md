### New file: `scripts/exclude_tinyusb_max3421.py`

A PlatformIO `pre:` extra-script that performs two build-tree adjustments to
installed lib_deps copies in `.pio/libdeps/<env>/` before the SCons builders
enumerate sources:

1. Writes a `library.json` into the Adafruit TinyUSB libdep with a
   `build.srcFilter` that excludes the entire tinyusb-core subtree
   (`class/`, `common/`, `device/`, `host/`, `osal/`, `portable/`,
   `tusb.c`). The Arduino-ESP32 framework's prebuilt `libarduino_tinyusb.a`
   already provides every one of those translation units; compiling them
   again from the Adafruit lib was producing wholesale multi-definition
   link errors. Only the Adafruit Arduino-glue layer under `arduino/` is
   compiled.
2. Patches the installed `BLE-MIDI/src/hardware/BLEMIDI_ESP32_NimBLE.h`
   to mark the out-of-class `begin(...)` definition `inline`. The library
   defines that body in a header without `inline`, producing the ODR
   violation on multi-TU includes. The upstream master fix templates the
   class — that is an API change requiring src/ updates which D10.1.1
   forbids — so Path E patches the libdep header instead.

Both adjustments are idempotent. Neither modifies files under `src/`,
neither forks upstream, neither uses `lib_ignore`.

## Why Paths A, B, C did not work

See `10.1-01-BLOCKER.md` for the full forensics. Brief summary:

- **Path A:** TinyUSB 2.4.3 is not in the registry (highest 2.x is 2.4.1,
  which still ships `hcd_max3421.c`). BLE-MIDI registry has been pruned to
  a single 2.2 (no 2.1 entry).
- **Path B:** `tusb_config_esp32.h` `#define`s `CFG_TUH_ENABLED 1` and
  `CFG_TUH_MAX3421 1` unconditionally (no `#ifndef` guard), so a
  command-line `-DCFG_TUH_ENABLED=0` is overridden by the header. The
  plan's Path B premise was incorrect.
- **Path C:** Pinning framework-arduinoespressif32 to a 3.20014 build
  fails because the current `espressif32` platform requires the new
  pioarduino build script that ships only in 3.20015+. Pinning the
  framework alone without the platform is structurally incompatible.

Path E was further broadened during execution: the original BLOCKER
identified only `hcd_max3421_*` multi-definition, but a clean rebuild
revealed the collision spans the entire tinyusb core (host + device +
class + portable). Path E excludes all of it; the T16 needs only the
prebuilt arduino_tinyusb (device-mode only on native DWC2).

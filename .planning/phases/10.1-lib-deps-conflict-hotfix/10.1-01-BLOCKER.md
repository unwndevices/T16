## Summary

All three resolution paths defined in `10.1-01-PLAN.md` Task 2 fail in practice
against the current upstream library / platform state. Plan-as-written cannot
be completed without scope expansion or a re-plan.

## What was tried

### Path A — pin to older minor versions (FAIL)

- **TinyUSB:** 2.4.3 does not exist on the PlatformIO registry. Highest 2.x is
  `2.4.1`. Tried `adafruit/Adafruit TinyUSB Library @ 2.4.1`. Link still
  fails — `hcd_max3421.c` is present in 2.4.1 and exports the same `hcd_*`
  symbols that collide with the prebuilt `libarduino_tinyusb.a`'s
  `hcd_dwc2.c.obj`.
- **BLE-MIDI:** `lathoub/BLE-MIDI` has only **one** version published on the
  PlatformIO registry — `2.2` (released 2022-01-20, queried via
  `https://api.registry.platformio.org/v3/packages/lathoub/library/BLE-MIDI`).
  No older registry version exists to pin to. Substituted upstream tag URL
  `https://github.com/lathoub/Arduino-BLE-MIDI.git#v2.1.0` (same upstream repo,
  not a fork) — that part is workable, but TinyUSB blocks the link before the
  BLE-MIDI half can be validated.

### Path B — keep TinyUSB 3.3.0, disable host stack via build flag (FAIL)

Added to `[env_common].build_flags`:
```
-DCFG_TUH_ENABLED=0
-DCFG_TUH_MAX3421=0
```

Link still fails with the same `hcd_*` multi-definition errors. Root cause:
the upstream file
`Adafruit TinyUSB Library/src/arduino/ports/esp32/tusb_config_esp32.h` lines
92 and 94 contain **unconditional** `#define`s:

```c
#define CFG_TUH_ENABLED 1
#define CFG_TUH_MAX3421 1
```

These are not guarded by `#ifndef`, so a `-DCFG_TUH_ENABLED=0` from `build_flags`
is overridden by the header at compile time. The plan's Path B assumes the
upstream guards honor command-line `-D` overrides — they do not.

### Path C — pin Arduino-ESP32 framework (FAIL)

Tried `platform_packages = platformio/framework-arduinoespressif32 @ 3.20014.231204`
(last 2.x-derived build, Dec 2023). Build aborts before link with:

```
*** missing SConscript file
'/home/unwn/.platformio/packages/framework-arduinoespressif32/tools/pioarduino-build.py'
```

The current `espressif32` platform (`55.3.37`) requires the new pioarduino build
script, which only ships in framework `3.20015+`. Pinning the framework alone
without also pinning the platform is structurally incompatible.

Additionally, Path C alone would fail Task 2's automated acceptance criterion
which requires `grep -E '(Adafruit TinyUSB Library @ 2\.|CFG_TUH_ENABLED=0)'`
to match — neither matches with a Path-C-only fix.

## Why this is a re-plan, not a "try harder"

The plan's three preference-ordered paths all hit upstream realities the plan
did not anticipate:

1. The PlatformIO registry has been pruned for `lathoub/BLE-MIDI` — only `2.2`
   remains. Path A's "pin BLE-MIDI to 2.1.0" is unreachable via the registry
   without falling back to a git URL (which is fine for this lib but not the
   plan's first-order suggestion).
2. The Adafruit TinyUSB `tusb_config_esp32.h` overrides command-line `-D`s
   unconditionally. Path B's "disable host stack via build flag" cannot work
   without also patching this header (which the plan forbids — "no source
   patches").
3. The arduino-esp32 platform now requires the pioarduino build script that
   only ships in 3.20015+, so the older 3.20014 framework cannot be paired
   with the current platform without ALSO pinning the platform itself
   (larger blast radius than Path C anticipated).

## What probably needs to happen next

A re-plan should consider one or more of:

- **Path D — pin the espressif32 platform AND framework to a known-good
  pre-host-stack version** (e.g. `platform = espressif32 @ 6.5.0` plus
  matching framework). Larger blast radius than Path C alone but
  structurally consistent.
- **Path E — `lib_filter`/`build_src_filter` on the TinyUSB lib to exclude
  `portable/analog/max3421/*`**. Not a `lib_ignore` (so doesn't silently
  drop the whole library) but a per-file exclusion. Plan currently scopes
  out only `lib_ignore` of the whole lib; per-file exclusion is arguably
  in-scope under "no functionality-disabling lib_ignore" since MAX3421
  host-controller support is not used by the T16 (it's USB device only).
- **Path F — replace the Adafruit TinyUSB lib_dep with an esp32-only
  packaging** (e.g. `adafruit/Adafruit TinyUSB Library` with a custom
  `build_src_filter` set on the env to exclude the analog/max3421 directory).
- **Path G — accept a controlled source patch under `src/` or a build script
  that strips the offending object** if the "no source patches" constraint
  is relaxed.

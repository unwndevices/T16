# Phase 10.1 — Lib_deps Conflict Hotfix: Context

**Phase goal:** Pin compatible third-party library versions so all four PlatformIO envs (`t16_debug`, `t16_release`, `t32_debug`, `t32_release`) link successfully, then run the deferred T16 hardware smoke test from Phase 10.
**Requirements:** TBD-LIB-01 (link parity), TBD-LIB-02 (smoke test parity)
**Inserted after:** Phase 10
**Reason:** Phase 10 surfaced two latent ODR/multi-definition errors in firmware lib_deps. Both predate v1.1 but were never caught because the v1.0 CI `firmware-build` job never executed on push (only the FTP-deploy workflow ran). The deferred T16 smoke test (ROADMAP success criterion #4 of Phase 10) cannot run without a flashable binary, and Phase 11 (HAL refactor) would build on top of broken linking.

---

## Locked Decisions

### D10.1.1 — Scope is link-only and version-pinning, not refactoring
- The fix MUST be expressed as version constraints in `platformio.ini` `lib_deps` (and/or `platform_packages` for the Arduino-ESP32 framework). No source patches in `src/`, no forks of upstream libraries, no `lib_ignore` workarounds that disable functionality.
- Acceptable: pinning to a known-good older minor version, switching to a different but officially-published version of the same library, switching the Arduino-ESP32 framework version.
- Not acceptable: vendoring third-party code, monkey-patching headers, using `#define` to silence symbols.

### D10.1.2 — Both link errors must be fixed in the same hotfix
- t16: TinyUSB host-controller multi-definition (Adafruit TinyUSB lib_dep `hcd_max3421_*` vs Arduino-ESP32 prebuilt `hcd_dwc2_*`).
- t32: BLE-MIDI 2.2.0 ODR violation (`BLEMIDI_ESP32_NimBLE::begin` defined in header without `inline`).
- Single hotfix phase produces both fixes; do not split across phases.

### D10.1.3 — Verify by full link, not by `pio check`
- Acceptance for the link fix is `pio run -e t16_release` AND `pio run -e t32_release` both producing `firmware.bin` (and `firmware.elf`) artifacts in `.pio/build/{env}/`.
- `pio run -e t16_debug` and `-e t32_debug` must also produce binaries to keep the CI matrix green end-to-end.

### D10.1.4 — T16 hardware smoke test runs in this phase, not deferred again
- The user has confirmed physical T16 hardware is at the bench.
- Once `t16_release` links, flash the binary and walk through the smoke checklist from `10-04-PLAN.md` Task 2:
  1. Startup animation reaches steady state.
  2. All 16 keys produce NoteOn/NoteOff via USB MIDI (verify in any DAW or via `tools/t16-tui`).
  3. Touch slider produces CC output.
  4. MODE button cycles modes (KEYBOARD → STRUM → XY_PAD → KEYBOARD).
  5. TOUCH button toggles touch mode.
  6. TRS MIDI output works on the renamed `TX2_PIN` GPIO.
  7. No behavioral regression vs. a known-good v1.0 binary.
- Smoke test result is recorded in `10.1-VERIFICATION.md` with the binary's git SHA and a per-criterion ✓/✗ table. If anything fails, file as a Phase 10 regression and gate Phase 11 on its fix.

### D10.1.5 — T32 hardware smoke test is NOT in scope here
- T32 hardware bring-up belongs to Phase 12. Phase 10.1 only requires the t32 binary to link cleanly; no T32 flashing is required.

### D10.1.6 — No new lib_deps, no behavioral feature changes
- Do not add new libraries, do not bump unrelated dependencies, do not change `build_flags` other than what is strictly required for the link fix.
- If a fix requires a `build_flags` change (e.g., `-DCFG_TUH_ENABLED=0` to disable an unused TinyUSB host-controller stack), that flag MUST be applied uniformly across all four envs and documented in the plan.

### D10.1.7 — Document the v1.0 latent-bug status in PROJECT.md
- Phase 10.1's SUMMARY must include a one-line note that this conflict was latent in v1.0 and not introduced by Phase 10. This protects the historical record and avoids Phase 10 being incorrectly blamed in future audits.

## Implementation Decisions (Claude's Discretion)

The specific resolution path is for the planner to determine — try in this preference order:

1. **Pin TinyUSB to a host-disabled version** (e.g., a 2.x release that does not unconditionally export `hcd_*`). Confirm via `pio pkg show "Adafruit TinyUSB Library"`.
2. **Disable TinyUSB host stack via build flag** (e.g., `-DCFG_TUH_ENABLED=0` if upstream supports it). Verify by checking for `hcd_*` symbols in the resulting `.elf`.
3. **Pin Arduino-ESP32 framework** to a version whose prebuilt `libarduino_tinyusb.a` does not include `hcd_*`, if such a version exists.
4. For BLE-MIDI: pin to a 2.x version where `begin()` is declared `inline` or only defined in a `.cpp` translation unit. If 2.2.0 is the latest, fall back to **2.1.x or earlier** that does not have the ODR violation.

The planner may try options out of order if research surfaces a known-good combination — the constraint is that whatever is chosen MUST be a published, downloadable version (not a fork or a private build).

## Existing Code Insights

- `platformio.ini` already has the four-env structure from Plan 10-01. lib_deps is in `[env_common]` and shared across all four envs.
- `-DUSE_TINYUSB` is set globally in `[env_common].build_flags`. Any TinyUSB stack-disabling flag added must coexist with this.
- The pre-Phase-10 single-env `[env:esp32s3]` used the same lib_deps; the conflict is environment-independent.
- v1.0 CI was `pio run -e esp32s3` then `pio test -e native`. The `firmware-build` job has no record of ever being green on `push` (only the FTP deploy workflow has run history per `gh run list`).

---

## Specific Ideas

- After link is green, before the hardware smoke test, run `pio run -e t16_release -t size` and compare against pre-Phase-10 baseline (commit `4b25496` or earlier) to ensure binary size hasn't ballooned (catches accidental enabling of a heavy alternate stack).
- Capture the `pio pkg list -e t16_release` output in the SUMMARY for both pre- and post-fix states so the version pin is auditable.

<objective>
Eliminate the two latent third-party-library link errors that block the v1.1 firmware build, then run the deferred T16 hardware smoke test. The fix is scoped strictly to `platformio.ini` `lib_deps` / `build_flags` changes — no source patches under `src/`, no forks, no functionality-disabling `lib_ignore` (per D10.1.1).

Two link errors must be resolved in the same hotfix (per D10.1.2):

1. **t16 link error:** Adafruit TinyUSB Library 3.3.0 lib_dep exports `hcd_max3421_*` host-controller symbols, and the Arduino-ESP32 prebuilt `libarduino_tinyusb.a` exports `hcd_dwc2_*` for the same symbol names — multi-definition at link time.
2. **t32 link error:** BLE-MIDI 2.2.0 defines `BLEMIDI_ESP32_NimBLE::begin(...)` in a header without `inline`, producing an ODR violation when more than one translation unit includes it.

Acceptance is full link (per D10.1.3): `pio run -e {t16,t32}_{debug,release}` all produce `firmware.bin` and `firmware.elf` in `.pio/build/{env}/`. After the link is green, the developer flashes the T16 release binary and walks the smoke checklist from Phase 10's `10-04-PLAN.md` Task 2 (per D10.1.4).

Purpose: Unblock the deferred T16 hardware smoke test (ROADMAP Phase 10 success criterion #4) and the entire Phase 11+ work that builds on a flashable firmware. Document that this conflict was latent in v1.0 (per D10.1.7) so Phase 10 is not retroactively blamed.

Output: A modified `platformio.ini` (lib_deps and/or build_flags only); four green firmware env binaries on disk; a `10.1-VERIFICATION.md` recording the smoke-test outcome with the binary's git SHA.
</objective>

**Resolution path A — preferred: pin to known-good older minor versions**

1. Replace the `Adafruit TinyUSB Library @ 3.3.0` entry. Try, in order:
   - `adafruit/Adafruit TinyUSB Library @ 2.4.3` (last stable 2.x — the host-controller symbols are conditional on `CFG_TUH_ENABLED` and not unconditionally exported)

   - `lathoub/BLE-MIDI @ 2.1.0` (the `begin()` ODR violation was introduced when BLE-MIDI moved the body into the header without `inline`; 2.1.0 predates that)

**Resolution path B — fallback: keep TinyUSB 3.3.0 and disable the host stack via build flag**

Only if Path A fails for TinyUSB:

1. Keep `adafruit/Adafruit TinyUSB Library @ 3.3.0`.
2. Add to `[env_common].build_flags` (uniformly applied to all four firmware envs):
   ```
   -DCFG_TUH_ENABLED=0
   -DCFG_TUH_MAX3421=0
   ```
   This is the upstream-supported way to disable the host-controller stack at compile time. It MUST coexist with the existing `-DUSE_TINYUSB`. Per D10.1.6, this build flag goes in `[env_common].build_flags` so it applies to all four envs identically.

6. **TRS MIDI output** works on the renamed `TX2_PIN` GPIO (verify with TRS-MIDI-to-USB adapter or downstream MIDI device).

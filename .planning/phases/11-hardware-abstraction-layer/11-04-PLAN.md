# Phase 11 — Plan 04: Configuration Layer Migration

## Goal

Remove the global `const uint8_t BANK_AMT = 4;` from `Configuration.hpp` (HAL-02 calls this out by name). Replace every consumer with `variant::CurrentVariant::kConfig.BANK_AMT`. Also confirm `Keyboard` and `DataManager` consume variant constants instead of literals (HAL-02 names them explicitly). T16 behavior unchanged.

**Note:** RESEARCH §1.2 / §4 closes Open Question A — `BANK_AMT` is invariant across T16/T32 (both are 4) but the success criterion text and HAL-02 wording require consumers to stop reading the global macro. Modeling it as a `kConfig.BANK_AMT` field keeps the migration consistent with the rest of HAL-02 even though the value never differs.

## Context

- RESEARCH §1.2, §1.3, §4: enumeration of `BANK_AMT` references and resolution of the "BANK_AMT is not actually variant-dependent" tension.
- CONTEXT D11.2: legacy macros deleted outright.
- HAL-02 (REQUIREMENTS.md line 20): `Adc`, `Keyboard`, `LedManager`, `DataManager` consume `HardwareVariantConfig` constants instead of macros.

### Task 4.3 — Audit Keyboard for variant-dependent literals

<read_first>
- src/Libs/Keyboard.hpp

<action>

```bash
grep -n "16\|TOTAL_KEYS\|key_amount" src/Libs/Keyboard.hpp src/Libs/Keyboard.cpp
```

Two outcomes:

a) `Keyboard` already takes `key_amount` as a runtime parameter (per `KeyboardConfig::Init(Key*, uint8_t key_amount)`). It does NOT hardcode 16. **No change needed.** Add a one-line comment in `Keyboard.hpp` near `KeyboardConfig`:

to make the design contract explicit and satisfy HAL-02's auditability.

</action>

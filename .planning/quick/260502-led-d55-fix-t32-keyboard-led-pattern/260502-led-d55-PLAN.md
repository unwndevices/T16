# Fix T32 keyboard-mode LED pattern

## Problem

On T32 (8×4 matrix, 32 keys), pressing a key in KEYBOARD mode produces no LED feedback. Two stacked root causes:

1. `src/services/InputProcessor.cpp:54` hard-codes `idx % 4` / `idx / 4` — wrong for an 8-wide matrix. For `idx ≥ 16`, `y` becomes 4..7, which `XY()` (`LedManager.cpp:47`) rejects, returning `0xFFFF`. `NoBlur` then writes to `patternleds[0xFFFF]` — out-of-bounds, corrupts adjacent memory.

2. `src/Libs/Leds/patterns/NoBlur.hpp` hard-codes a 4×4 grid:
   - `GenerateLut()`: `lumaleds[16]`, `blur2d(..., 4, 4, ...)`, `luma[10][16]`
   - `RunPattern()`: `for (x=0..3) for (y=0..3) { ...luma[step][abs(x-pos_x)+abs(y-pos_y)*4]; patternleds[XY(x,y)] |= ... }` — only writes left half of T32, and the LUT stride is fixed at 4.

## Tasks

### T1 — Fix key→position mapping (InputProcessor.cpp)

- File: `src/services/InputProcessor.cpp`
- Action: Replace hard-coded `idx % 4` / `idx / 4` at line 54 with variant-aware width. Use `variant::CurrentVariant::kConfig.MATRIX_WIDTH` (already exposed via `LedManager.hpp` as `kMatrixWidth`).
- Verify: T16 still maps idx 0..15 to (x: 0..3, y: 0..3). T32 maps idx 0..31 to (x: 0..7, y: 0..3).
- Done: Edit applied, file compiles.

### T2 — Generalize NoBlur to variant matrix dimensions

- File: `src/Libs/Leds/patterns/NoBlur.hpp`
- Action:
  - Size `lumaleds[]` and `luma[10][...]` to `kMatrixSize` (= width × height).
  - Build LUT with `blur2d(lumaleds, kMatrixWidth, kMatrixHeight, ...)`.
  - Iterate `for (x=0..kMatrixWidth-1) for (y=0..kMatrixHeight-1)`; index LUT with stride `kMatrixWidth`: `luma[step][abs(x-pos_x) + abs(y-pos_y) * kMatrixWidth]`.
  - The LUT growth needs care: `blur2d` strength constants (`2 + j*4`) were tuned for 4×4 — keep them for now; visual fidelity tuning is a follow-up if needed.
- Verify: T16 visual output unchanged (same 4×4 LUT footprint when matrix is 4×4). T32 produces a halo on key presses across the full 8×4.
- Done: Edit applied, file compiles.

- Action: `pio run -e t16_debug` and `pio run -e t32_debug`.
- Done: Both succeed with no new warnings related to NoBlur/InputProcessor.
